# GitHub 관리 규칙

MacKey 저장소의 커밋, 이슈, 브랜치, PR 작성 규칙. 지금까지의 작업 이력을 기준으로 정리했다.

## 커밋 컨벤션

[Conventional Commits](https://www.conventionalcommits.org/) 기반, 제목과 본문은 한국어로 작성한다.

### 형식

```
<타입>: <제목>

<본문 (선택)>
```

### 타입

| 타입 | 용도 | 예시 |
|---|---|---|
| `feat` | 새 기능 추가 | `feat: 설정창 단축어 드래그 앤 드롭 순서 변경` |
| `fix` | 버그 수정 | `fix: 패널 초기 높이 콜백이 무시되는 순서 버그 수정` |
| `docs` | 문서만 변경 | `docs: README 빌드 섹션 갱신` |
| `build` | 빌드 스크립트, 패키지 설정 | `build: .app 번들 생성 스크립트 추가` |
| `chore` | 그 외 잡무 (gitignore 등) | `chore: 빌드 산출물 gitignore 추가` |
| `refactor` | 동작 변화 없는 구조 개선 | `refactor: 파서 분리` |

### 규칙

- 제목은 50자 내외, 마침표 없이 명사형 또는 서술형으로 끝낸다 ("~수정", "~추가", "~변경")
- 본문은 **무엇을/왜** 중심으로 작성한다. 특히 `fix`는 원인을 본문에 남긴다
  (예: "contentView 설정 시점에 SwiftUI 초기 레이아웃이 동기 실행되어 …")
- 여러 변경 사항은 `-` 목록으로 나열한다
- 하나의 커밋에는 하나의 관심사만 담는다 (파서 변경과 UI 변경은 분리)

## 이슈 발행 규칙

### 제목

```
[타입] 제목
```

- 타입: `Feat`, `Fix`, `Docs` 등 커밋 타입의 대문자 표기. 복합이면 `[Fix/Feat]`
- 예: `[Feat] 단축어 관리 기능 및 패널 크기 자동 조절`

### 본문 템플릿

```markdown
## 개요
무엇을 왜 하는지 1~3줄

## 작업 내용
- [ ] 체크박스로 작업 단위 나열
- [ ] 완료 기준이 판별 가능한 수준으로 구체적으로

## 참고 (선택)
- 관련 문서, 포맷, 제약 사항
```

## 브랜치 규칙

```
<타입>/<kebab-case-설명>
```

- 예: `feat/shortcut-viewer`, `feat/reorder-and-height-fix`, `docs/github-conventions`
- 이슈 하나당 브랜치 하나, `main`에서 분기한다
- 머지된 브랜치는 삭제한다

## PR 규칙

### 제목

```
[타입] 제목 (#이슈번호)
```

- 예: `[Fix/Feat] 패널 높이 미조절 재수정 및 드래그 순서 변경 (#5)`

### 본문 템플릿

```markdown
## 개요
Closes #이슈번호

변경 목적 1~2줄

## 주요 변경
- **fix**: 원인과 해결 방식 요약
- **feat**: 추가된 동작 요약

## 테스트
- 검증한 방법과 결과 (수동 확인 항목 포함)

## 비고 (선택)
- 리뷰어(미래의 나)가 알아야 할 맥락
```

### 머지

- `Merge pull request` (머지 커밋) 방식 사용
- 머지 전 로컬에서 빌드·동작 확인을 마친다
- `Closes #N`으로 이슈가 자동으로 닫히는지 확인한다
