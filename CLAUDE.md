## 프로젝트 개요

**Runlet**(가칭) — 터미널에서 명령어를 쳐야 실행되던 작업을, 메뉴바 드롭다운 버튼 하나로 실행하는 개인용 macOS 명령어 런처.

자주 쓰는 셸 명령(예: `KeyFlow` 빌드+실행)을 "이름 붙은 버튼"으로 등록해두고 메뉴바에서 한 번 클릭으로 백그라운드 실행한다.

> `Runlet`은 가칭이며 교체 가능. `KeyFlow`는 이 앱이 아니라 이 런처가 실행시킬 별개의 Mac 앱이다.

## 확정된 핵심 명세 (임의 변경 금지)

| 항목 | 확정 내용 |
|---|---|
| 앱 껍데기 | **메뉴바 앱** (`MenuBarExtra`) — 아이콘 클릭 시 명령 리스트 드롭다운 |
| 명령 관리 | 여러 개 등록/수정/삭제 가능. 편집은 **별도 관리 창**, 드롭다운은 **실행 전용** |
| 실행 방식 | 클릭 시 **백그라운드에서 조용히 실행** (별도 터미널 창 없음) |
| 실행 피드백 | **없음** (무피드백, 조용히 실행만) |
| 데이터 모델 | `RunCommand` = `이름` + `여러 줄 셸 스크립트` (작업 디렉토리는 스크립트 내 `cd`) |
| 실행 엔진 | `Process`로 로그인 셸 `["/bin/zsh", "-l", "-c", 스크립트전체]` 에 전달 |
| 샌드박스 | **반드시 OFF** (임의 셸/바이너리 실행에 필수) |
| 배포 | 개인용 로컬 앱 (Mac App Store 아님, 서명·공증 불필요) |
| 최소 버전 | macOS 13 (Ventura) 이상 |

**비목표:** 터미널 에뮬레이터, 실행 히스토리/로그 뷰어, SSH·스케줄링, 다중 계정/동기화, App Store·샌드박스 대응.

## 기술 스택

- 언어/UI: **Swift + SwiftUI** (`MenuBarExtra`, 관리 창)
- 셸 실행: Foundation `Process`
- 영속화: `Codable` + JSON — `~/Library/Application Support/Runlet/commands.json`
- (선택, M5) 로그인 항목: `ServiceManagement` `SMAppService`

**주의점**
- 샌드박스 OFF 전제 — 프로젝트 설정에 Sandbox capability를 넣지 않는다.
- **PATH:** GUI 앱 프로세스는 셸 PATH를 자동 상속하지 않는다 → `-l`(로그인 셸)로 실행해 `.zprofile`/`.zshrc`의 PATH(Homebrew `/opt/homebrew/bin` 등)를 물려받는다.
- 여러 줄·셸 문법 스크립트는 인자를 쪼개지 말고 `-c`로 스크립트 전체를 zsh에 넘겨 해석시킨다.

## 아키텍처 (의존 방향: 위 → 아래 한 방향)

```
[UI]       MenuBarListView (드롭다운) · ManagerWindowView (관리 창)
              ▼
[서비스]   CommandStore (목록 상태·JSON 저장/로드) · CommandRunner (셸 실행)
              ▼
[모델]     RunCommand (id + name + script)
```

- `CommandRunner`는 UI·저장소를 모른다 (스크립트 문자열만 받아 실행).
- `CommandStore`는 실행 로직을 모른다 (데이터만 관리).
- 드롭다운과 관리 창은 같은 `CommandStore` 상태를 공유해 편집이 즉시 반영된다.

## 로드맵 (관찰 가능한 동작으로 검증)

- **M0** 빈 메뉴바 앱 셋업 (macOS 13, 샌드박스 OFF)
- **M1** 하드코딩된 KeyFlow 명령 1개 실행 + `CommandRunner` 초안
- **M2** `RunCommand` 모델 + `CommandStore` + JSON 영속화
- **M3** 관리 창 CRUD (추가/수정/삭제 즉시 반영)
- **M4** 실행 견고화 (로그인 셸/PATH, UI 비블로킹 백그라운드 실행)
- **M5** 마감 (아이콘, 옵션 로그인 자동 실행)

## 협업 규칙 (바이브코딩)

1. **자율 구현.** 코드 이해나 리뷰를 사용자에게 요구하지 않고 끝까지 완성한다.
2. **보고는 관찰 가능한 동작 중심.** 내부 클래스/함수가 아니라 "무엇이 되는지 / 어떻게 켜서 확인하는지 / 어떤 모습인지"로 보고한다.
3. **마일스톤 단위 진행.** 각 M을 끝낼 때마다 확인 방법을 함께 제시한다.
4. **확정 명세 임의 변경 금지.** 바꿀 이유가 보이면 트레이드오프와 함께 합의를 먼저 받는다.
5. **위임된 결정은 단정하지 않는다.** 합리적 기본값으로 진행하되 무엇을 왜 골랐는지 보고한다.
6. **추측한 API를 넣지 않는다.** 불확실한 API/시그니처는 Apple 공식 문서로 확인하고, 버전 한정 API는 가용성을 명시한다.
7. **용어는 Swift/SwiftUI/AppKit로.** 다른 스택 비유를 쓰지 않는다.

## Git 규칙 (요약 — 상세는 CONVENTIONS.md)

- **커밋:** Conventional Commits 기반, 한국어. `<타입>: <제목>` (타입: `feat`/`fix`/`docs`/`build`/`chore`/`refactor`). 제목 50자 내외·마침표 없음. `fix`는 본문에 원인을 남긴다. 커밋 하나에 관심사 하나.
- **브랜치:** `<타입>/<kebab-case-설명>`, `main`에서 분기, 이슈당 하나, 머지 후 삭제.
- **이슈:** 제목 `[타입] 제목`. 본문 = 개요 / 작업 내용(체크박스) / 참고.
- **PR:** 제목 `[타입] 제목 (#이슈번호)`. 본문에 `Closes #N`. 머지 커밋 방식, 머지 전 로컬 빌드·동작 확인.
- PR은 사용자가 GitHub 웹 UI에서 직접 만든다 (`gh pr create` 사용 안 함).
